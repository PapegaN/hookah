# Архитектура Hookah

## Цель

Собрать резюме-проект для управления кальянной: вести каталог табака, контролировать остатки, оформлять заказы и постепенно расширять систему до полноценного back-office продукта.

## Репозитории

### `hookah-web`

Отвечает за интерфейс оператора, администратора и в будущем гостевые сценарии.

Ключевые зоны:

- дашборд смены;
- каталог табака;
- учет остатков и движений;
- заказы и статусы исполнения;
- справочники оборудования и рецептов.

### `hookah-api`

Отвечает за бизнес-логику, авторизацию, валидацию, интеграции и публикацию API-контракта.

Ключевые bounded contexts:

- `auth`;
- `catalog`;
- `equipment`;
- `recipes`;
- `inventory`;
- `orders`.

### `hookah-db`

Источник истины для схемы PostgreSQL, миграций, сидов и локального окружения базы данных.

Схема БД на текущем этапе разделяется на доменные пространства:

- `auth`;
- `catalog`;
- `equipment`;
- `recipes`;
- `inventory`;
- `sales`;
- `support`.

## Поток данных

1. `hookah-db` задает структуру таблиц, ограничений и enum-типов.
2. `hookah-api` работает поверх PostgreSQL и публикует REST API.
3. `hookah-web` использует API как единственный способ чтения и записи данных.

## Инфраструктурный слой

- Главный репозиторий содержит `compose.yaml` для совместного запуска `web`, `api` и `db`.
- Каждый прикладной репозиторий обязан иметь собственный `Dockerfile`, чтобы модуль можно было собрать отдельно.
- Для локального старта всей системы frontend проксирует `/api` на backend-контейнер, а backend подключается к PostgreSQL по внутреннему имени сервиса `db`.

## Стратегия строгой типизации

- На фронтенде все доменные объекты описываются явными TypeScript-интерфейсами.
- На backend DTO и ответы API описываются классами и типами, пригодными для Swagger/OpenAPI.
- Контракт между frontend и backend в будущем можно автоматизировать через генерацию типов из OpenAPI.
- Изменения схемы БД проходят через версионируемые SQL-миграции.
- Ключевые доменные ограничения по возможности проверяются на уровне БД, а не только в приложении.

## Начальные функциональные контуры

- Каталог табака: бренды, линейки, вкусы, профили и оценочные характеристики.
- Оборудование: чашки, кальяны, калауды и уголь.
- Забивки: состав смеси, оборудование, прогрев и текстовый отзыв.
- Учет остатков: партии, движения и доступный вес.
- Заказы: заказ, позиции заказа, статус и сумма.

## Нефункциональные требования

- Читаемая модульная архитектура.
- Подготовка под CI с линтингом, тестами и проверкой сборки.
- Осмысленная документация, чтобы проект было не стыдно показать в резюме и на собеседовании.
