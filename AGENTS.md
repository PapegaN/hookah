# Правила работы по проекту Hookah

## Контекст

Главный репозиторий содержит orchestration-слой и общую документацию.

Прикладная часть разбита на три отдельных репозитория-сабмодуля:

- `hookah-web` — фронтенд на Vue 3.
- `hookah-api` — backend на NestJS.
- `hookah-db` — схема PostgreSQL, миграции и сиды.

## Структура superproject

- Корень проекта хранит `compose.yaml`, архитектурные документы и ссылки на сабмодули.
- Изменения в коде прикладных частей коммитятся внутри соответствующего сабмодуля.
- После коммита в сабмодуле отдельно коммитим обновленный указатель в главном репозитории.

## Базовые принципы

- Держим границы между репозиториями явными: UI, API и БД развиваются независимо.
- Считаем API основным контрактом между фронтендом и backend.
- Выбираем строгую типизацию по умолчанию и избегаем `any`, если можно выразить доменную модель типами.
- Архитектурные решения фиксируем в документации вместе с мотивацией.

## Технологический стек

- `hookah-web`: Vue 3, Composition API, TypeScript, Vite, Vue Router, Pinia, Vitest, Playwright, ESLint, Prettier.
- `hookah-api`: NestJS, TypeScript strict mode, REST API, Swagger/OpenAPI, class-validator, ESLint, Prettier, Jest.
- `hookah-db`: PostgreSQL, SQL-миграции, сиды, Docker image для локальной БД.
- Корневой репозиторий: Docker Compose для совместного запуска всей системы.

## Правила коммитов

- Все коммиты пишем на русском языке.
- Один коммит — одно логическое изменение.
- Заголовок коммита должен объяснять результат, а не техническое действие ради действия.
- Предпочтительный стиль заголовка: глагол в неопределенной форме или повелительной форме.
- Если изменение нетривиальное, добавляем тело коммита с тремя блоками:
  - зачем внесено изменение;
  - что именно сделано;
  - как это было проверено.

## Минимальный чеклист перед коммитом

- Обновлена документация, если поменялась архитектура или правила интеграции.
- Прогнаны релевантные проверки для затронутого репозитория.
- В коммит не попали случайные форматирования и несвязанные правки.
- Если менялся сабмодуль, в главном репозитории отдельно зафиксирован новый gitlink.

## Ожидания по качеству

- Имена сущностей должны совпадать между UI, API и БД.
- Новые модули должны быть готовы к масштабированию, а не только к демо-сценарию.
- Если решение принято как временное, это явно фиксируется в README или ADR-документе.
